# Plan: Move files context from system prompt to `list_files` tool

## Problem

The system prompt includes a dynamic `files_section` listing delivered and pending
files. This section sits between cached system blocks and messages:

```
[GLOBAL_PROMPT (cached 1h)] + [files_section (NOT cached)] + [messages...]
```

When `deliver_file` adds/removes files, `files_section` changes. Since Anthropic
caching is prefix-based, this shifts the entire message prefix, causing a cache
miss on ALL conversation messages (~10K+ tokens re-read at full price instead of
0.1x cached price).

## Solution

Replace dynamic files_section with a static `list_files` tool. Claude calls it
on-demand to get the current file list. System prompt becomes fully static.

```
[GLOBAL_PROMPT (cached 1h)] + [custom_prompt (cached 1h)] → STABLE
[tools including list_files] → STABLE (definitions never change)
[messages with rolling 5m breakpoint] → STABLE cache prefix
```

## Architecture

### New tool: `list_files`

- **Type:** client-side tool (like deliver_file, render_latex)
- **Cost:** Free (only DB/cache lookup, no external API)
- **Input:** No required parameters (thread_id injected by executor)
- **Output:** Formatted list of delivered + pending files with file_ids

### Data flow

```
User message → Handler loads files from DB/cache (as before)
            → But does NOT put them in system prompt
            → Passes files data to tool executor context
            → Claude decides whether to call list_files
            → Tool returns files list as tool_result
```

## Implementation Steps

### Step 1: Create `bot/core/tools/list_files.py`

New tool module following existing patterns (deliver_file.py as template):

```python
TOOL_CONFIG = ToolConfig(
    name="list_files",
    definition=LIST_FILES_TOOL,
    executor=list_files,
    emoji="📋",
    needs_bot_session=True,  # needs session for DB queries
)
```

Tool definition (static, cached with other tools):
```python
LIST_FILES_TOOL = {
    "name": "list_files",
    "description": "List all files available in the current conversation. "
        "Returns delivered files (images, PDFs, audio, documents sent by user "
        "or generated by you) and pending files (generated but not yet delivered). "
        "Call this when the user references attachments, previously sent files, "
        "or generated content. Also call before using analyze_image, analyze_pdf, "
        "transcribe_audio, or deliver_file to get the correct file_id/temp_id.",
    "input_schema": {
        "type": "object",
        "properties": {},
        "required": []
    }
}
```

Executor function:
- Receives `bot`, `session`, `thread_id` (standard for needs_bot_session tools)
- Queries `UserFileRepository.get_by_thread_id(thread_id)` for delivered files
- Queries `get_pending_files_for_thread(thread_id)` for pending files
- Returns formatted result using existing `format_unified_files_section()`
- If no files: returns `{"files": "none", "message": "No files in this conversation"}`

### Step 2: Register tool in `bot/core/tools/registry.py`

- Import `TOOL_CONFIG as LIST_FILES_CONFIG` from `core.tools.list_files`
- Add `"list_files": LIST_FILES_CONFIG` to `TOOLS` dict
- No other changes needed (get_tool_definitions auto-includes it)

## Risk Assessment

**Low risk:**
- Static system prompt = always cache-compatible
- Tool is free (no balance check needed)
- Existing tools unaffected

**Medium risk:**
- Claude might not always call list_files when needed
  → Mitigated by clear hint in system prompt + tool description
- Extra round-trip adds ~3-5s latency when files are checked
  → Only happens when Claude decides files are relevant
  → Most messages don't involve files

**Rollback:** Revert to files_section in system prompt (simple git revert).

## Expected Impact

- Cache stability: message-level cache survives deliver_file operations
- Cost savings: ~$0.10-0.15 per cache miss avoided (at Opus pricing)
- Latency: slight increase (+3-5s) only on file-related messages
- Net: significant savings for conversations with many messages after file delivery
